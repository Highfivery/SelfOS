## 1. Overview

A **questionnaire** is a structured set of questions a subject (the **sender**) sends to a **recipient**
— another household person, or an external person with no app — to learn something that helps the coach
support the sender better. The feature is four loosely-coupled systems behind one name, kept separate so
each can grow independently:

1. **Authoring** — a questionnaire is created fresh each time: pick a **type**, then any mix of (a) your
   **own** questions, (b) a **free-text brief** AI turns into questions, and (c) AI-generated questions
   **from your latest data** (recent sessions, prior answers, Insights…). **There are no pre-built
   templates.**
2. **Generation / personalization** — AI reads the sender's existing **structured context** via a
   pluggable **context-provider registry** (profiles, relationships, prior answers, Insights — and any
   future feature that registers a provider) to draft, tailor, de-duplicate, and refresh questions.
3. **Delivery & collection** — get the questionnaire to the recipient (in-app inbox for household people;
   a **zero-knowledge relay link** for external people) and get answers back.
4. **Analysis & assimilation** — turn raw answers into structured **Insights** (and chartable
   **metrics**) that, once approved, feed `buildContext` so every future coaching session is better
   grounded.

The questionnaire **schema is delivery- and recipient-agnostic**, so channels grow without touching
authoring, generation, or analysis. Raw answers are always **encrypted**; what flows into the coach is
the **derived Insight**, never the raw answers wholesale.

### 1.1 The shared Insight / metrics layer

`Insight` is a single, source-discriminated record (`source: 'questionnaire' | 'session' | 'dream'`) stored
encrypted per person/relationship — the app's **long-term memory** (the thing `05` deferred). It carries
free-text facts **and** a `metrics` map of named, normalized numeric signals (e.g. `connection`,
`desire`, `moodValence`) that are **chartable over time**. Questionnaires define and own this layer here;
`09` (session analysis), `11` (tracking dashboard), and `12` (dreams) build on it. Consumers —
`buildContext`, the gap-finder, the "what the coach knows" surface, and `11`'s trends — read it regardless
of source.

## 2. Goals / Non-goals

**Goals**

- A **template-free, delivery-agnostic** questionnaire model: `Questionnaire → Assignment → ResponseSet`,
  with an extensible **answer-type** library, **question media** (author-attached images), and **simple
  conditional branching**.
- **Create = type + own questions + AI**: a **type** taxonomy (starter list **+ custom**) that steers
  generation/categorization; AI generation from a **free-text brief** and/or the sender's **structured
  context** via a **context-provider registry**; **preview / edit / test-on-self** before sending.
- **In-app delivery** (household inbox) **and** a **per-household, zero-knowledge Cloudflare relay** for
  external recipients, provisioned + deployed **from the app** (Settings).
- **Privacy by design**: a **private mode** (break-glass, §8); a **configurable compatibility mode** whose
  visibility setting **derives the recipient's disclosure** so the app can never over-promise.
- **Analysis → Insight (+ metrics) → `buildContext`** with an **approve-step**, the shareable-vs-private
  split, recency/relevance **prioritization + a per-person token cap**, full **edit/delete + provenance**.
- A **gap-finder** ("Suggested for you") that proposes the next questionnaire/questions from structured
  context (via the same registry).
- **Full metering**: every AI call (`questionnaire.generate` / `.suggest` / `.analyze`) is budget-gated
  and recorded through `06`.
- **Sensitive content in scope**: Intimacy tiers up to **Explicit / Unfiltered**, behind **tiered age +
  consent gates**, operated **strictly within Anthropic's usage policy** (graceful refusal handling).
- The **shared Insight / metrics layer** that `09` and `11` build on.

**Non-goals (deferred / owned elsewhere)**

- **Relationship/intimacy dashboard, metric trends across check-ins, and the intimacy/encounter log** —
  owned by [`11-relationship-tracking.md`](11-relationship-tracking.md); 08 provides the metrics
  _mechanism_, not the dashboards.
- **Session analysis** — companion [`09-session-analysis.md`](09-session-analysis.md); 08 defines the
  shared Insight/metrics layer it produces into.
- **360° / multi-recipient aggregation** — v1 is one recipient per send (model stays per-recipient, a
  non-breaking later add).
- **Recurring / scheduled check-ins + reminders** — deferred; the schema is **re-ask-ready** (re-sends are
  manual), so a scheduler is a later, additive slice.
- **Conversational / adaptive answering** — v1 is forms + simple branching everywhere.
- **Respondent file/photo upload** — out of scope (author-attached question images are in; respondent
  uploads are not, to keep the relay privacy surface small).
- **Broad in-app internationalization** — v1 app UI is English; the **relay page is localizable**.

`questionnaires.answer`) and the route tree `/questionnaires`:

> **Build state:** the **Questionnaires** nav/route are gated by **`questionnaires.create`**; a separate
> **Inbox** nav/route (`/inbox`, with an unanswered badge) is gated by **`questionnaires.answer`** (built
> 2026-06-11, [§13](#13-build-slices) slice 5 / §13.5a). The sender-side **Results** view + the live
> Analyze trigger + `questionnaires.autoAnalyze` are built (2026-06-11, §13.5b); per-question trends +
> deletion/purge are built (2026-06-11, §13.5c); compatibility (variants + alignment + the 3 visibility
> modes + the explicit-grant `readRaw` capability) is built (2026-06-11, §13.5d; the audit log was later
> removed 2026-06-14 — the Owner is now the full-access role, see [`04`](04-people-roles.md) §8).

- **My Questionnaires** — the sender's created questionnaires (drafts + sent history), each **re-sendable**
  (for trends). _(No template library — questionnaires are created fresh.)_
- **Builder** — create/edit; pick a type; add own questions and/or AI-generate; preview; **test-on-self**;
  send.
- **Inbox** — questionnaires assigned to the **active person** (household recipients), with a badge.
- **Results & Insights** — per send, per person, per relationship; per-question **rating-over-time** across
  re-asks and the **compatibility alignment** view. _(The aggregated relationship/intimacy dashboard +
  metric trends live in [`11`](11-relationship-tracking.md).)_
- **Suggested** — the gap-finder.

Admin-only surfaces (relay setup, the disclosure toggle, raw-access audit log, any cost figure / the
Everyone scope) carry the standard **"Admin only"** marker (CLAUDE.md §12). Every screen is responsive
(~360px→desktop) and the relay page is independently responsive + WCAG-compliant.

### 3.1 Authoring (no templates: type + own questions + AI)

1. **Pick a type** — from the starter taxonomy (_Fill the gaps_, _How am I doing in this role?_, _Honest
   outside view / blind spots_, _Intimacy_ [General · Adventurous · Explicit · Unfiltered], _Perspective
   on a recent event_, _Appreciation / strengths / weaknesses_, _Science-informed_, _Scenario_ families
   [Sexual · Relationship · Friendship · Work · Family]) **or a custom type** the user names. The type
   steers generation + categorizes trends; it carries **no pre-authored content**.
2. **Add questions, any mix of:**
   - **Your own** — add/reorder/duplicate; pick answer types (§4.3); required/optional; **attach an image**
     to a question; add **simple branching** ("show Q/section if answer to Qx = value").
   - **A free-text brief** — "what do you want to dive into?" → AI drafts questions from it.
   - **From your data** — AI generates from the **context-provider registry** (profiles, relationships,
     prior answers, Insights, recent sessions once `09` ships), **avoiding already-answered items** and
     **flagging stale data** worth refreshing. Output is schema-validated; a **safety pass** runs.
   - Per-question AI assists ("reword warmer", "make this for [person]").
3. **Preview / Test-on-self** — see exactly what the recipient sees; optionally answer it yourself as a dry
   run (does **not** create coaching Insights).

A sent questionnaire is an **immutable snapshot** (§4.2): editing a created questionnaire makes a new
version; in-flight assignments keep their original questions so answers always map.

### 3.2 Sending

1. **Choose recipient** — a household **subject** (delivered in-app) or an **external person** (relay
   link). External requires `questionnaires.sendExternal` (default ON for Member) **and** a configured
   relay (§3.8); without one, the sender is prompted to connect it (in-app/household sending still works).
2. **Privacy mode** — _Standard_ (sender sees answers) or **Private** (break-glass, §8). For
   **compatibility** questionnaires, choose a **visibility mode** (§3.6) — this **drives the recipient's
   disclosure text** so the promise is always honest.
3. **Sender identity (external)** — **named by default**, with a per-send **anonymous** option.
4. **Sensitive gating** — Intimacy _General_ → **18+ acknowledgement**; **Explicit / Unfiltered** →
   **date-of-birth gate + explicit consent** on the recipient side (§8.3).
5. **Link + delivery (external)** — the app generates the relay link + a **6-digit PIN**, and offers
   **Copy link**, a prefilled **`mailto:`** (subject + body unique to this send; recipient email prefilled
   from the Person record), a prefilled **SMS** (`sms:` on mobile), and the **native share sheet**. The
   **PIN is included in the email + SMS by default** (per-send opt-out for sensitive sends). Default wording
   is the **`questionnaires.defaultMessages`** templates, editable in Settings (`{sender}`/`{link}`
   placeholders) and tweakable per send.

### 3.3 Recipient — in-app (household)

Appears in the recipient's **Inbox** when they're the active person. They answer (save/resume), see the
**purpose, who's asking, the privacy mode, and a clear "Decline"** (silently, or with an optional short
note), then submit. Editable until submit, then locked.

### 3.4 Recipient — external (relay page)

A standalone, branded, responsive, WCAG page served by the household's relay Worker (no app needed):

1. **Open link → PIN gate.** The page requests the PIN before releasing any question content (the relay
   serves only ciphertext; the browser decrypts with the key in the URL **fragment**, which never reaches
   the server). Rate-limited (5 attempts → 15-min lockout). Author-attached **question images** are
   likewise served as ciphertext and decrypted client-side (§8.6).
2. **Consent / disclosure.** Shows purpose, **who's asking** (or "Someone" if anonymous), the **privacy
   statement derived from the configured visibility**, the **age gate** for sensitive tiers, the minimal
   **privacy notice**, **always-visible crisis resources + the not-medical line**, a quiet **"Sent securely
   via SelfOS"** trust line, and a clear **Decline** (silently, or with an optional short note).
3. **Answer.** Same renderer as in-app (one renderer, two hosts). Long forms **save + resume** (link +
   PIN). Answers are encrypted **in the browser** to the send's public key and POSTed as ciphertext.
4. **Return visit.** After submit, the link shows **"Thanks for filling this out."** A **withdraw** link
   deletes the response **before** it's drained; there is **no post-submit answer review** (SelfOS keeps no
   plaintext copy — zero-knowledge). Expired/revoked links show **"This questionnaire is no longer
   available."**

### 3.5 Collection / drain

The app **drains** responses on open (and via a manual "Check for responses"); a "new responses" indicator
surfaces. Each is decrypted locally with the send's private key (§5), validated, and the relay copy is
deleted on successful drain. Multi-device drains are idempotent.

### 3.6 Compatibility mode

Both people answer a per-answerer **personalized variant** of the same underlying questions. The author
**writes the canonical questions once**; AI **generates each answerer's personalized variant**, aligned by a
**canonical question id** (§4.3), so the comparison stays apples-to-apples. AI then produces an alignment
report. The author picks a **visibility mode** per questionnaire — and the recipient's disclosure follows it
automatically:

- **Shared report, raw hidden both ways** (default) → recipient told _"Private…"_.
- **Sender sees all** → recipient told _"Your answers will be shared with [sender]"_ (requires
  `questionnaires.readRaw`, §8).
- **Each sees own + shared report** → recipient told their answers stay private to them; both see the
  joint report.

### 3.7 Analysis, Insights, metrics & the gap-finder

- **Analyze** — analysis is **manual by default**: the sender taps **Analyze** on a response (a setting
  `questionnaires.autoAnalyze`, default **OFF**, can run it automatically on arrival). It's budget-gated
  (§6) and turns answers into a structured **Insight**: a summary, key **facts**, and any **metrics** the
  questions declared (§4.3 `metricKey`); a **crisis flag** is raised if answers disclose risk (§8.2).
- **Approve-step** — before the Insight enters `buildContext`, the sender reviews it in one pass: **edit the
  text**, **choose which facts are shareable** (§4.4), or discard. Approved Insights stay editable/deletable
  later. Raw answers are never sent wholesale to the coach.
- **Insights surface ("what the coach knows")** — list/edit/delete Insights with **provenance**; the
  shareable-vs-private split (`04`) governs whether a fact about a related person can feed **that** person's
  coaching.
- **Per-question trends (built §13.5c)** — for numeric questions (rating/slider, and per row/bucket for
  matrix/allocation) the Results view charts each recipient's answers **over re-asks** (≥2 points), via core
  `buildQuestionTrends` + the `assignments:trends` IPC + the `LineChart` primitive. Trends include
  **Private** sends' numeric values (the disclosure is worded accordingly, §3.2/§8.4) — numbers only, never
  the prose answers.
- **Suggested (gap-finder)** — AI scans **structured context only** (via the context-provider registry —
  profiles, relationships, prior answers + Insights; **never raw chat transcripts**) and proposes the next
  questionnaire/questions. Budget-gated; over budget it renders a calm "budget reached" state without
  calling Claude.

### 3.8 Relay setup (admin-only Settings → Relay)

Per-household **bring-your-own Cloudflare**, provisioned + deployed from the app:

1. **Connect** — paste a **least-privilege Cloudflare API token** (Workers + KV/D1/R2 deploy scopes) +
   account id. The app **provisions** storage + **deploys** the zero-knowledge relay Worker to a free
   **`*.workers.dev`** subdomain (custom domain optional, later).
2. **Manage** — status, **one-click update** when SelfOS ships a new relay version, and **teardown** on
   disable/uninstall.
3. **Config storage** — endpoint, drain secret, and the **encrypted Cloudflare token** live in the **vault**
   (`config/relay.enc`, §4), so any household member sends/collects with **zero setup** and any admin
   device manages without re-entering the token.

### 3.9 Deletion

> **Build state (built §13.5c):** core `purgeQuestionnaire` (def + every send's snapshot/assignment/
> response + derived Insights — and, for a compatibility send, the `questionnaires/compat/<groupId>/
report.enc` + the group's derived Insight, §13.5d) + `deleteSend` + `hasSends`; `Questionnaire.creatorPersonId` is main-stamped
> on create (preserved on edit, never back-filled onto a legacy def). `questionnaires:delete` is role-aware
> — the Owner (`people.manage`) purges at any stage; a non-owner **creator** deletes their own
> **only while unsent**. Per-send delete is `assignments:delete` (sender/admin-only). Inline "Are you sure?"
> confirms in the builder + each Results send card. **Built §13.6:** the relay-link **revoke** before an
> external send's teardown, and **question-image garbage collection** — `purgeQuestionnaire`/`deleteSend`
> (and an image-dropping save) run `imageGc.garbageCollectImages`, which deletes media referenced by no
> live definition or send snapshot (an image still frozen in a sent snapshot is kept until that send is
> also gone).

- **Owner** deletes **any** questionnaire at any stage → **purge everything** (questionnaire +
  relay link + raw responses + Insights) after a clear confirmation; the relay link is revoked.
- **Creator** (incl. a Member) deletes **their own** questionnaire **only while unsent**, which also revokes
  any preview relay link.

## 4. Data model

All persisted formats are **Zod-backed** (`z.infer` types), versioned (`schemaVersion` + migrations), and
written through the vault service (`00` §4). Private content is encrypted (AES-256-GCM, master key — `04`
§5). Types live in `@selfos/core` so the renderer, IPC contract, and relay package share one source.

### 4.1 Vault layout (additions)

```
vault/
  config/
    relay.enc                      # encrypted: relay endpoint + drain secret + Cloudflare token
    questionnaires.json            # plain: default message templates, custom types, non-secret prefs
  questionnaires/
    defs/<questionnaire-id>.enc    # a created Questionnaire (versioned; re-sendable)
    media/<image-id>.enc           # encrypted author-attached question image (shared dir; isMediaPath-guarded)
    sends/<assignment-id>/
      questionnaire.enc            # immutable snapshot of the sent Questionnaire
      assignment.enc               # Assignment (status, channel, privacy, recipient, keys)
      response.enc                 # ResponseSet (encrypted answers) — present once collected
      consent.enc                  # ConsentReceipt (external)
    compat/<group-id>/
      report.enc                   # AlignmentReport for a compatibility group (§3.6/§13.5d)
  people/<person-id>/insights/<insight-id>.enc          # Insight (source: questionnaire|session)
  relationships/<rel-id>/insights/<insight-id>.enc      # relationship-scoped Insight
```

**Multi-device / single-vault.** All questionnaire data — created definitions, sends, responses, Insights,
and the relay config (incl. the encrypted Cloudflare token) — lives **in the vault**, so every install
pointing at the same folder shares it; this feature stores **no per-device state**. Device-local `userData`
holds only the cross-cutting items owned by `00`/`04` (the keychain master key, window/session +
active-person state, logs). The **Owner is the full-access role** — raw-answer access is permitted to the
Owner (any Private send) or a `senderSeesAll` sender holding `questionnaires.readRaw`; the previous
break-glass audit log was removed 2026-06-14 (per [`04-people-roles.md`](04-people-roles.md) §8).

### 4.2 Questionnaire & Question

```ts
type AnswerType =
  | 'shortText'
  | 'longText'
  | 'singleChoice'
  | 'multiChoice'
  | 'rating'
  | 'slider'
  | 'ranking'
  | 'thisOrThat'
  | 'yesNo'
  | 'date'
  | 'matrix'
  | 'allocation';

interface BranchRule {
  // simple conditional only (v1)
  whenQuestionId: string;
  equals: string | number | boolean;
  action: 'show'; // show this question/section when the rule matches
}

interface Question {
  id: string; // stable within a questionnaire
  canonicalId?: string; // shared across personalized variants (compatibility alignment)
  type: AnswerType;
  prompt: string;
  help?: string;
  required: boolean;
  media?: { imagePath: string; alt: string; mime: string }; // author-attached image (encrypted; ZK on relay)
  options?: string[]; // choice/ranking/thisOrThat/allocation buckets
  scale?: { min: number; max: number; minLabel?: string; maxLabel?: string; step?: number };
  matrix?: {
    rows: string[];
    min: number;
    max: number;
    minLabel?: string;
    midLabel?: string;
    maxLabel?: string;
    pointLabels?: string[]; // N-point labelled scale (length === max−min+1); wins over min/mid/maxLabel
    limitLabels?: string[]; // subset of pointLabels rendered with a boundary/limit tone (e.g. ['Hard no'])
  };
  metricKey?: string; // rating/slider/matrix → populates Insight.metrics[metricKey] (§4.4)
  branch?: BranchRule;
}

type SensitivityTier = 'standard' | 'intimacyGeneral' | 'explicit' | 'unfiltered';

interface Questionnaire {
  id: string;
  schemaVersion: number;
  version: number; // immutable-snapshot version
  creatorPersonId?: string; // main-stamped author (on create only); gates "creator deletes own while unsent" (§3.9); legacy defs lack it → Owner-deletable-only
  title: string;
  description?: string;
  type: string; // a starter-taxonomy key OR a user-defined custom type
  sensitivity: SensitivityTier;
  questions: Question[];
  compatibility?: { enabled: true; visibility: 'sharedReport' | 'senderSeesAll' | 'eachSeesOwn' };
  createdAt: string;
  updatedAt: string;
}
```

`allocation` questions use `options` as the buckets (answers must sum to 100). `matrix` rates each
`matrix.rows` item on the one shared scale. Custom types are free-text strings persisted in
`questionnaires.json` so they reappear in the type picker.

```ts
}

```

### 4.4 Insight (the shared Insight / metrics layer)

```ts
type InsightSource = 'questionnaire' | 'session' | 'dream'; // 'session' by spec 09, 'dream' by spec 12

interface InsightFact {
  id: string;
  text: string;
  shareable: boolean; // false = private to the subject; true = may feed related people
}

interface Insight {
  id: string;
  schemaVersion: number;
  source: InsightSource;
  subjectPersonId: string; // whose coaching this informs
  relationshipId?: string; // relationship-scoped insights
  summary: string;
  facts: InsightFact[];
  metrics?: Record<string, number>; // named normalized signals (chartable; e.g. connection, desire,
  //   moodValence, moodEnergy). The flexible basis for all trends.
  confidence: 'low' | 'medium' | 'high';
  approved: boolean; // questionnaire insights require approval; session insights auto-true (09)
  provenance: {
    assignmentId?: string;
    conversationId?: string;
    dreamId?: string; // spec 12
    compatibilityGroupId?: string; // §13.5d: a compatibility alignment Insight
    at: string;
  };
  crisisFlag?: boolean;
  createdAt: string;
  updatedAt: string;
}
```

`metrics` is the single, extensible basis for every trend in the app: questionnaire answers populate it via
`Question.metricKey`; `09` adds `moodValence`/`moodEnergy`; `11` defines the intimacy/relationship metric
vocabulary and the dashboards. Adding a metric is data, not a schema change. `buildContext` (in
`@selfos/core/people`) is extended to append the active person's **approved** Insights + **shareable** facts
from related people's Insights, **prioritized by recency + relevance and capped by a per-person token
budget**.

### 4.5 ConsentReceipt, RelayConfig, Settings & Person additions

```ts

  // config/relay.enc (encrypted)
```

## 5. Architecture & modules

### 5.1 Core (`@selfos/core`) — platform-agnostic

- **insightStore** — encrypted CRUD over `Insight`; `query`, `summarizeForContext` (prioritize + cap),
  `queryMetrics` (for trends). The shared Insight/metrics layer; consumed by `buildContext`, `09`, `11`.
- **questionnaireService** — encrypted CRUD over created definitions; immutable-snapshot versioning.
- **customTypeService** — persists user-defined custom types to the plain `config/questionnaires.json`
  prefs file (`QuestionnairePrefsSchema`), so they reappear in the builder's type picker on every device.
- **assignmentService / responseService** — lifecycle, status transitions, ResponseSet round-trip, re-ask
  chaining.
- **answering** — pure, DOM-free answering logic shared by every host that renders a questionnaire to be
  answered (preview/test-on-self now; Inbox + relay later): `isQuestionVisible`/`visibleQuestions` (live
  branching), `isAnswered`/`unansweredRequired` (required gating), `allocationTotal`.
- **imageService** — encrypted CRUD over author-attached question images in the shared media dir
  (`storeQuestionnaireImage`/`getQuestionnaireImage`/`deleteQuestionnaireImage`; `isMediaPath` guard;
  `ALLOWED_IMAGE_MIME`/`MAX_IMAGE_BYTES`), over the byte-level `encryptBytes`/`decryptBytes` crypto.
- **contextProviderRegistry** — `registerContextProvider({ id, gather(personId|relId) })`; generation +
  the gap-finder pull from all registered providers (profiles, relationships, prior answers, Insights; `09`
  registers a session-insight provider; future features register their own). **This is the extensibility
  backbone** — new context sources need no generator changes.
- **generationService / analysisService / gapFinderService** — build prompts from the registry, call the
  `ClaudeClient` host, **safety-pass** + schema-validate output, emit Insights/metrics; all budget-gated.
- **promptBuilder (extended)** — generation/analysis/gap prompts; `cache_control` on the stable prefix;
  **adaptive thinking** on analysis.
- **relay crypto** — per-send keypair, content-key/PIN handling, response + image sealing — reuses
  `@selfos/core/crypto` (WebCrypto + scrypt).

### 5.2 Desktop main (host)

- **cloudflareDeployer** — provisions KV/D1/R2 + deploys/updates/tears down the Worker via the Cloudflare
  REST API; writes `RelayConfig`. (iOS uses the same REST calls — feasibility flagged §11.)
- **relayClient** — POST/drain/revoke against the deployed Worker (drain authed by `drainSecret`).
- Wires core services to `nodeFileSystem` / `nodeSecretStore` / `anthropicClient` and registers IPC.

### 5.3 Renderer

- **Stores (Zustand):** `questionnaireStore`, `assignmentStore`, `insightStore`, `relayStore`.
- **Screens:** My Questionnaires, Builder (+ answer-type editors incl. matrix/allocation + question-image
  attach, AI-generate, preview, test-on-self), Inbox, Results & Insights (+ rating-over-time +
  compatibility alignment → `/gallery` chart primitives), Suggested, and the admin **Relay setup** panel.
- **Shared answering renderer** — `QuestionnaireForm` (built) renders the 12 answer-type controls + live
  branching/required (over the core `answering` helper) + the crisis footer; it's used by preview/test-on-self
  now and the in-app Inbox next, and is **physically extracted to a shared package when the relay page (§5.4)
  needs it** so both hosts share one implementation.

### 5.4 `apps/relay` (Cloudflare Worker)

A zero-knowledge mailbox + static answering page. Stores **only ciphertext** keyed by `token`; gates content
(and question-image) release by `pinHash` (rate-limited); accepts size-bounded (≤256 KB) well-formed
response payloads for known tokens; serves the answering page (bundling the shared renderer + core crypto);
authenticates **drain/revoke** by `drainSecret`; minimal logging; serves the minimal **privacy notice** and
**crisis resources** statically.

### 5.5 Feature-module registration

Standard `defineFeature` (`00` §5.2): nav + routes + the `questionnaires.*` capabilities + the new vault
schemas + IPC handlers + the Settings declarations + the context-provider registrations. The shell is
untouched.

## 7. States & edge cases

Per `00` §7, every surface handles loading / empty / error / offline. Specifically:

- **No Claude key / AI disabled** — manual authoring + sending still work; generation/analysis/gap-finder
  show calm "enable AI" states; no Insights produced.
- **Over budget** — generation/analysis/gap-finder block with the `06` warn→block UX (owner override); the
  gap-finder degrades to "budget reached" without calling Claude.
- **No relay configured** — in-app/household sends work; external send is offered with a "connect a relay"
  prompt; never an error.
- **Relay deploy failure / token invalid / quota** — typed, actionable errors; partial provisioning is
  idempotently retried; nothing half-written claims success.
- **PIN brute-force** — lockout after 5 attempts/15 min; the sender can reset/resend the PIN.
- **Expired / revoked / already-submitted link** — neutral "no longer available" / "thanks" states.
- **Lost device before drain** — responses recoverable (the send private key is wrapped under the master
  key in the vault, recovery-phrase recoverable).
- **Edited/deleted definition with in-flight sends** — sends use their immutable snapshot; unaffected.
- **Sync conflict** on a questionnaire/Insight/relay file — vault conflict detection (`00`); never
  auto-deleted; the `relay.enc` token conflict is surfaced, not overwritten.
- **Schema migration** — `Person` (email/phone) + any questionnaire/Insight bumps migrate on read.
- **Crisis content in a free-text answer** — analysis raises `crisisFlag`; the result **leads with
  resources** (§8.2).

## 8. Safety, privacy & honesty

### 8.1 Wellness boundary

SelfOS remains **wellness/self-help, not medical** (CLAUDE.md §1). Questionnaires never diagnose, score
diagnostically, or reproduce copyrighted/clinical instruments — "science-informed" means **original,
evidence-informed** questions in the SelfOS voice. The not-medical line is visible in-app and on the relay.

### 8.2 Crisis routing

The relay page and the Inbox always show **curated crisis resources + "Get help now"** (static; no Claude on
the relay). During analysis a model-based **crisis flag** (no keyword interstitial, consistent with `05`)
makes the result **lead with concern + resources**. For external recipients SelfOS can't intervene live —
the always-present static resources are the safeguard.

### 8.3 Sensitive content & age

Intimacy _General_ → **18+ acknowledgement**; **Explicit / Unfiltered** → **date-of-birth gate + explicit
consent** before any explicit content (incl. question images) renders; a **ConsentReceipt** is stored with
the response; household recipients use the birthday on their Person record where present. All
generation/analysis runs **within Anthropic's usage policy** — refusals handled gracefully, never
circumvented. **iOS/App-Store risk** for explicit tiers is recorded (§11) with mitigations (17+ rating, the
gate, store-guideline compliance, remote disable).

### 8.4 The private-mode privacy model

- **Promise to the recipient:** in **Private** mode the recipient is told their answers are _"used only to
  personalize coaching."_ In normal operation that is true: **no UI exposes the raw/prose answers**, and the
  `questionnaires.readRaw` capability ships **OFF for non-owner roles**. _Since §13.5c the disclosure also
  states that the recipient's **numeric ratings may appear in the sender's trends over time** (the
  rating-over-time charts include Private sends' numbers — never the prose answers); the copy is kept
  consistent across the send panel + the recipient Inbox so the promise stays honest (§3.2)._
  `questionnaires.readRaw` is an **explicit-grant-only** capability: it ships OFF for non-owner roles and is
  enabled only by an explicit toggle in the Roles matrix. The **Owner is the full-access role and always
  holds it.**
- **Raw answers can be revealed two ways** (each via `assignments:revealRaw`, which writes **no audit
  entry** — the break-glass audit log was removed 2026-06-14): the **Owner** (any Private send), or the
  **sender of a `senderSeesAll` compatibility send** holding `questionnaires.readRaw` (§3.6/§13.5d).
  Nothing else exposes the prose answers.
- **We never surface owner/admin visibility to answerers** (a durable product rule, 2026-06-15). The
  `questionnaires.discloseAdminAccess` setting + the `ADMIN_ACCESS_DISCLOSURE` line were **removed** (§16.6):
  no recipient is ever told a household owner/administrator could access their answers, and no author-facing
  copy mentions it either.
- **Honesty guard** — the recipient's disclosure text is **derived from the configured visibility/privacy
  mode**, so no configuration can promise more privacy than the system delivers. (It states what the
  **sender** sees — never owner/admin reach.)
- **Not zero-knowledge from the owner** — consistent with `04` §8 and `10-multi-device-vault` §2/§8 (one
  master key decrypts the whole vault; RBAC is enforced at the app/UX layer, not by encryption).

### 8.5 Coercion & consent

Every send shows the recipient the **purpose, who's asking, the privacy mode, and a frictionless Decline**;
sensitive sends emphasize they may skip/stop; no dark patterns. External recipients can **withdraw/delete**
before drain.

### 8.6 Relay & data handling

Zero-knowledge: the Worker stores **ciphertext only**; questions **and author-attached images** are
decrypted via the URL **fragment** key (never seen by the server); responses are encrypted to the send public
key in the browser. A **minimal privacy notice** (what's stored, that it's encrypted + unreadable to the
host, retention, how to delete) is served on the relay — required despite zero-knowledge because
IP/timestamps/payloads are still processed (GDPR/CCPA + App Store). The Cloudflare token is **least-privilege**
and **encrypted** in the vault.

## 15. 2026-06 amendment — authoring-experience refinements

Layers on the built feature (§1–§14 remain accurate). Covers app-refresh items **6** (General type +
intimacy/scenario-only sensitivity), **7** (clearer "who sees what" copy), **8** (drop the "Use my information" toggle),
and **9** (live inline question preview). Mostly renderer; the only schema-adjacent change is one taxonomy
entry and where sensitivity is surfaced. No new capabilities, no new IPC channels.

### 15.1 A "General" type (item 6)

Add **`general`** ("General") to the starter taxonomy (§3.1) — a catch-all for a questionnaire that isn't tied
to a specific frame (role-feedback, blind-spots, intimacy, …). It becomes the **default selected type** for a
new questionnaire (today the builder defaults to whatever is first; General is the most neutral starting point).
Like every type it carries **no pre-authored content** and simply steers generation/categorization. No schema
change — `Questionnaire.type` is already a free string keyed off the starter list + custom types; `general` joins
the starter list (single source of truth shared by the type `<select>` and the gap-finder).

### 15.2 Sensitivity is intimacy-only (item 6)

Today the **Sensitivity** picker shows the same four intimacy-flavoured options
(`standard` / `intimacyGeneral` / `explicit` / `unfiltered`) on **every** type, which reads as out of place on,
say, a role-feedback questionnaire. New rule:

A small **`SENSITIVITY_TYPES`** constant (single source of truth) declares which types surface the picker, with
two behaviours:

- **`intimacy`** — always sensitive. The picker shows the **intimacy tiers only**
  (`intimacyGeneral` · `explicit` · `unfiltered`), default `intimacyGeneral`; **no `standard` option** (an
  intimacy questionnaire is at least 18+ general). Switching **to** intimacy seeds `intimacyGeneral`.
- **`scenario`** — _may_ be sensitive (a scenario can be sexual, or entirely tame — work, family). The picker
  shows with **`standard` as the default** plus the intimacy tiers available to escalate, so a sexual scenario
  can be marked explicit while a work scenario stays standard.
- **All other types** — sensitivity is **implicitly `standard`**, the picker is **hidden**, and the value is
  forced to `standard` on save. Switching a questionnaire's type to one **not** in `SENSITIVITY_TYPES` resets
  sensitivity to `standard`.

The `SensitivityTier` enum (§4.2) is **unchanged**; this is purely _when_ the control is shown and what default
it carries. All the existing send-time age/consent gating (§8.3) is unaffected — it already keys off the tier,
which only `intimacy`/`scenario` questionnaires can now carry as non-`standard`.

### 15.3 Clearer "who sees what" copy (item 7)

The **compatibility visibility** picker (§3.6) keeps its three modes (`sharedReport` / `eachSeesOwn` /
`senderSeesAll`) and the whole privacy/audit mechanism (§8.4) — only the **author-facing labels + help** change,
to drop jargon the user flagged ("break-glass", "audited", the redundant-sounding "each sees their own"):

| mode            | new label                            | new help (author-facing)                                                                                                                    |
| --------------- | ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------- |
| `sharedReport`  | **Shared report only**               | "Neither of you sees the other's answers — you both get one combined report."                                                               |
| `eachSeesOwn`   | **Shared report + your own answers** | "You both get the combined report, and each person can also look back at their **own** answers. Neither sees the other's."                  |
| `senderSeesAll` | **You see their answers**            | "You'll see [recipient]'s individual answers and you both get the combined report. They're clearly told their answers are shared with you." |

When `senderSeesAll` is selected, a plain secondary line is shown to the author. _(Amended 2026-06-14: now that
the audit log is removed (§8.4), this line reads "You'll be able to read their raw answers (a household owner can
too) — let them know." — honest about who can see the answers, with no audit-trail language.)_

Notes:

- "Each sees their own" → **"Shared report + your own answers"** makes the _added_ thing explicit (you can review
  your own responses), removing the "obviously I see my own" redundancy.
- **"break-glass" / "audited" disappear from user copy.** The `senderSeesAll` reveal still requires

- The recipient-facing **disclosure** text (`compatibilityDisclosure`, §3.6) is already plain and honesty-derived;

### 15.4 Remove the "Use my information" AI toggle (item 8)

In the **Draft-with-AI** panel (§3.1), the **"Use my information"** toggle is **removed**. The author's own
shareable data is **always** used to personalise generation (that's the whole point — highly personalised
questions). The panel keeps the _other_ context controls that are genuinely a choice: when a **target person** is
selected, "Use [person]'s shareable info" and "Use our relationship" remain (they pull a _different_ person's
context, the §13.3 shareable-only boundary). `gatherGenerationContext` always includes the author profile; the
removed toggle's `includeAuthor` flag is dropped (it was effectively always-on already).

### 15.5 Live inline question preview (item 9)

As the author edits a question — its prompt, answer type, options/scale/matrix, help — a **live preview** of how
that single question will render to a recipient appears **inline**, updating in real time. It reuses the shared

This complements — doesn't replace — the existing whole-questionnaire **Preview / Test-on-self** mode (§3.1): the
inline preview is per-question while authoring; the full mode is the end-to-end recipient experience.

## 16. 2026-06-15 amendment — audit fixes & enhancements

A full audit of the built feature (the compatibility send flow, the authoring workflow, AI generation by
tier, and test coverage) found three real problems plus the three enhancements the user asked for. This
section specifies all of them. Layers on §1–§15 (which remain accurate). All decisions were asked-first.

### 16.1 Compatibility participant model — the sender can be a participant (the core fix)

**Problem.** Compatibility today assumes the **sender is a neutral third party** comparing **two other**
people (the bridge rejects the sender being a participant — "Send a compatibility check to two other
people."). But the obvious couples case is "compare **me** with my partner." A user authoring "Sexy Time"
and picking themselves + their partner hits a confusing late rejection, and the disclosure copy
(`compatibilityDisclosure(visibility, senderName)`) names the **sender** as one of the two answerers
("neither you nor [sender]…"), which is also shown to the recipients — wrong on both counts.

**Resolution (decision: "you + someone else", keep two-others).** Compatibility supports **both**:

- **You + someone else** (the primary, default case): the **sender is one of the two participants**; the
  other person answers their personalized variant; the report/insight follows the visibility mode.

### 16.2 Context-only visibility mode (new) — no report, feeds both contexts

**Decision.** Add a **fourth** compatibility visibility option,
`CompatibilityVisibility += 'contextOnly'`: both people answer their variants, but **no report is generated
or shown to anyone and no raw answers are shared** — the answers are distilled into Insight(s) that

- **Analysis** — `alignmentService` (or a sibling) runs the distillation but produces **no `AlignmentReport`

- **Picker** — `contextOnly` joins the §15.3 "who sees what" list with plain copy: _"No report — answers

### 16.3 Create → Send: a clear two-step workflow

**Problem.** The builder shows **Create/Save** and **Send** side-by-side, both gated by the same `canSave`,
both terminal — but "Create" saves **and closes the builder** (so you can't Send after it) while "Send"
**auto-saves internally** then opens the send panel. "Create then Send" looks like a sequence but isn't.

**Resolution (decision: "Save draft → Send from the saved questionnaire").**

- While editing, the primary action is **Save** (creates/updates the draft and **keeps you on the saved
  questionnaire**, not closing the builder). A clear saved/dirty state distinguishes "new draft" from
  "saved."
- **Send** is a distinct step available **on a saved questionnaire** (its own button on the saved
  detail/list row), not a co-equal of Save in the edit footer. Sending still validates first; if the draft
  has unsaved edits it prompts/saves them. This makes the flow read **create → then send**.
- Remove the strand where Create closes the builder and leaves no path to Send.

### 17.3 Recipient-bound questionnaires (the core model change)

- **Every questionnaire targets exactly ONE recipient. Never several.** (Compatibility is the **only**
  exception — see §17.5.)
- **Recipient is chosen FIRST** — step 1 of creating any questionnaire, before authoring or AI drafting (the
  way compatibility already picks participants first). You cannot author or save a questionnaire "for nobody."
- The recipient is **a household person OR an external (relay) recipient**. The existing relay recipient
  details (name/contact) move from the send panel to the create step for external recipients.
- **Duplicate to another person** — because a questionnaire is bound to one recipient, re-asking someone else
  is a new questionnaire. A **"Duplicate"** action clones an existing questionnaire's questions into a new one
  where you pick the new recipient (and which re-runs de-dup for that new person if AI-drafted).
- **Flow:** pick recipient → author (manual and/or "Draft with AI") → **Send** still dispatches (creates the
  in-app `Assignment` or mints the relay link). The §16.3 two-step stays, but the **recipient picker leaves
  the send panel** (it's now set at creation); Send becomes channel/privacy confirmation + dispatch.
- **Recipient vs. "about a person" context (RESOLVED).** The **recipient** is who _answers_; the existing
  **"about a specific person"** toggle is AI _context_. Selecting a household recipient **defaults the "about"
  context to that person**, but it stays **overridable** (clear it, or point it at a third party) — one mental
  model: the recipient is the subject unless you say otherwise. De-dup always keys off the **recipient**.

### 17.4 Recipient-aware generation (de-dup) — full content, host-side, author-blind

- When the recipient is a **household person**, generation gathers that person's **full answered content**:
  their **onboarding intake** (incl. restricted facts), **Session** transcripts/insights, **prior
  questionnaire** questions+answers, and **profile + approved insights** — assembled **host-side** and sent to
  Claude **only** as de-dup grounding.
- **Deliberate privacy-rule relaxation (confirmed).** This crosses the otherwise-strict own-context-only rule
  (a person's private/restricted material informing another person's action). It is allowed **only** for this
  generation path, and **only** because: (a) it runs **host-side in main**, (b) the **author never sees the raw
  content** — only the resulting questions come back, and (c) the model is instructed to **avoid** overlapping
  topics, **never to reference, echo, or reveal** what the recipient disclosed. **Safety requirement
  (non-negotiable):** the prompt must forbid the model from quoting or alluding to the recipient's prior
  content in any generated question — de-dup means _steer clear of_, not _mention_. CLAUDE.md §1's "never
  surface private data to others" is preserved at the **output** boundary (questions reveal nothing); only the
  model's transient input is relaxed.
- **External recipients** have no household history → **skip de-dup** (the prompt just gets the brief + the
  author's own context + intimacy inventory, as today).
- **Avoid adjacency, not just exact dups** — the prompt instructs the model to skip both already-answered
  questions and closely-related ones (same topic already covered).
- **Manual (non-AI) authoring (RESOLVED).** The **recipient requirement** applies to manual authoring too, but
  automated de-dup is an **AI-generation** feature — we can't rewrite hand-typed questions, and surfacing "they
  already answered this privately" to the author would leak. So manual authoring **binds a recipient** and
  unlocks the AI de-dup, but adds **no overlap warning** in v1 (no scaffolding).

contextOnly). They are **exempt** from the one-recipient rule and keep their existing participant model and
variant/alignment machinery unchanged. The "exactly one recipient" rule governs every **other** type.

### 17.10 Critical fix — intimacy generation "No usable questions" was a thinking-budget bug, NOT a refusal

After §17.2 shipped, intimacy `explicit`/`unfiltered` generation **still** returned "No usable questions came
back" **every time** (General worked). Diagnosed against the **live** API (the offline fake always returns
questions, masking it): the model was **NOT refusing**. `anthropicClient.stream` (the app's real path) sets
`thinking: { type: 'adaptive' }`, and **`max_tokens` is the COMBINED thinking + output budget**. For the long,
sensitive intimacy prompt, adaptive thinking on **sonnet** (the default model) consumed the entire 1500-token
budget → `stop_reason: max_tokens` with **empty output** → `extractJsonObject` → null → REFUSED. (Opus thought
less and fit; General's prompt is short so thinking stayed small — which is why only intimacy/sonnet failed.)

**Fix:** structured one-shot JSON calls disable extended thinking. `ClaudeStreamOptions` gains
`extendedThinking?: boolean` (default on — chat/coaching keep adaptive thinking); the Electron + iOS clients
omit the `thinking` field when it's `false`; `runClaude` (questionnaire generate / improve / variant /
gap-finder) passes `extendedThinking: false`, and generation's budget is raised to **2500** for long
multi-choice option lists. **Diagnostic:** a cut-off/empty reply now reports "The AI's draft was cut off…
Please try again" distinctly from a genuine no-JSON refusal. **Verified live** (sonnet, both tiers, thinking
off → `end_turn` + valid JSON). **Lesson: adaptive `thinking` shares the `max_tokens` budget — a bounded
structured-JSON call must disable it (or reserve a large budget), or heavy thinking silently truncates the
output to empty; and a symptom that looks like a content refusal can be pure token starvation, so diagnose
against the LIVE model (the offline fake hides it) before touching the prompt.**
