## 1. Overview

A **questionnaire** is a structured set of questions a subject (the **sender**) sends to a **recipient**
‚Äî another household person, or an external person with no app ‚Äî to learn something that helps the coach
support the sender better. The feature is four loosely-coupled systems behind one name, kept separate so
each can grow independently:

1. **Authoring** ‚Äî a questionnaire is created fresh each time: pick a **type**, then any mix of (a) your
   **own** questions, (b) a **free-text brief** AI turns into questions, and (c) AI-generated questions
   **from your latest data** (recent sessions, prior answers, Insights‚Ä¶). **There are no pre-built
   templates.**
2. **Generation / personalization** ‚Äî AI reads the sender's existing **structured context** via a
   pluggable **context-provider registry** (profiles, relationships, prior answers, Insights ‚Äî and any
   future feature that registers a provider) to draft, tailor, de-duplicate, and refresh questions.
3. **Delivery & collection** ‚Äî get the questionnaire to the recipient (in-app inbox for household people;
   a **zero-knowledge relay link** for external people) and get answers back.
4. **Analysis & assimilation** ‚Äî turn raw answers into structured **Insights** (and chartable
   **metrics**) that, once approved, feed `buildContext` so every future coaching session is better
   grounded.

The questionnaire **schema is delivery- and recipient-agnostic**, so channels grow without touching
authoring, generation, or analysis. Raw answers are always **encrypted**; what flows into the coach is
the **derived Insight**, never the raw answers wholesale.

### 1.1 The shared Insight / metrics layer

`Insight` is a single, source-discriminated record (`source: 'questionnaire' | 'session' | 'dream'`) stored
encrypted per person/relationship ‚Äî the app's **long-term memory** (the thing `05` deferred). It carries
free-text facts **and** a `metrics` map of named, normalized numeric signals (e.g. `connection`,
`desire`, `moodValence`) that are **chartable over time**. Questionnaires define and own this layer here;
`09` (session analysis), `11` (tracking dashboard), and `12` (dreams) build on it. Consumers ‚Äî
`buildContext`, the gap-finder, the "what the coach knows" surface, and `11`'s trends ‚Äî read it regardless
of source.

## 2. Goals / Non-goals

**Goals**

- A **template-free, delivery-agnostic** questionnaire model: `Questionnaire ‚Üí Assignment ‚Üí ResponseSet`,
  with an extensible **answer-type** library, **question media** (author-attached images), and **simple
  conditional branching**.
- **Create = type + own questions + AI**: a **type** taxonomy (starter list **+ custom**) that steers
  generation/categorization; AI generation from a **free-text brief** and/or the sender's **structured
  context** via a **context-provider registry**; **preview / edit / test-on-self** before sending.
- **In-app delivery** (household inbox) **and** a **per-household, zero-knowledge Cloudflare relay** for
  external recipients, provisioned + deployed **from the app** (Settings).
- **Privacy by design**: a **private mode** (break-glass, ¬ß8); a **configurable compatibility mode** whose
  visibility setting **derives the recipient's disclosure** so the app can never over-promise.
- **Analysis ‚Üí Insight (+ metrics) ‚Üí `buildContext`** with an **approve-step**, the shareable-vs-private
  split, recency/relevance **prioritization + a per-person token cap**, full **edit/delete + provenance**.
- A **gap-finder** ("Suggested for you") that proposes the next questionnaire/questions from structured
  context (via the same registry).
- **Full metering**: every AI call (`questionnaire.generate` / `.suggest` / `.analyze`) is budget-gated
  and recorded through `06`.
- **Sensitive content in scope**: Intimacy tiers up to **Explicit / Unfiltered**, behind **tiered age +
  consent gates**, operated **strictly within Anthropic's usage policy** (graceful refusal handling).
- The **shared Insight / metrics layer** that `09` and `11` build on.

**Non-goals (deferred / owned elsewhere)**

- **Relationship/intimacy dashboard, metric trends across check-ins, and the intimacy/encounter log** ‚Äî
  owned by [`11-relationship-tracking.md`](11-relationship-tracking.md); 08 provides the metrics
  _mechanism_, not the dashboards.
- **Session analysis** ‚Äî companion [`09-session-analysis.md`](09-session-analysis.md); 08 defines the
  shared Insight/metrics layer it produces into.
- **360¬∞ / multi-recipient aggregation** ‚Äî v1 is one recipient per send (model stays per-recipient, a
  non-breaking later add).
- **Recurring / scheduled check-ins + reminders** ‚Äî deferred; the schema is **re-ask-ready** (re-sends are
  manual), so a scheduler is a later, additive slice.
- **Conversational / adaptive answering** ‚Äî v1 is forms + simple branching everywhere.
- **Respondent file/photo upload** ‚Äî out of scope (author-attached question images are in; respondent
  uploads are not, to keep the relay privacy surface small).
- **Broad in-app internationalization** ‚Äî v1 app UI is English; the **relay page is localizable**.

- **My Questionnaires** ‚Äî the sender's created questionnaires (drafts + sent history), each **re-sendable**
  (for trends). _(No template library ‚Äî questionnaires are created fresh.)_
- **Builder** ‚Äî create/edit; pick a type; add own questions and/or AI-generate; preview; **test-on-self**;
  send.
- **Inbox** ‚Äî questionnaires assigned to the **active person** (household recipients), with a badge.
- **Results & Insights** ‚Äî per send, per person, per relationship; per-question **rating-over-time** across
  re-asks and the **compatibility alignment** view. _(The aggregated relationship/intimacy dashboard +
  metric trends live in [`11`](11-relationship-tracking.md).)_
- **Suggested** ‚Äî the gap-finder.

Admin-only surfaces (relay setup, the disclosure toggle, raw-access audit log, any cost figure / the
Everyone scope) carry the standard **"Admin only"** marker (CLAUDE.md ¬ß12). Every screen is responsive
(~360px‚Üídesktop) and the relay page is independently responsive + WCAG-compliant.

### 3.1 Authoring (no templates: type + own questions + AI)

1. **Pick a type** ‚Äî from the starter taxonomy (_Fill the gaps_, _How am I doing in this role?_, _Honest
   outside view / blind spots_, _Intimacy_ [General ¬∑ Adventurous ¬∑ Explicit ¬∑ Unfiltered], _Perspective
   on a recent event_, _Appreciation / strengths / weaknesses_, _Science-informed_, _Scenario_ families
   [Sexual ¬∑ Relationship ¬∑ Friendship ¬∑ Work ¬∑ Family]) **or a custom type** the user names. The type
   steers generation + categorizes trends; it carries **no pre-authored content**.
2. **Add questions, any mix of:**
   - **Your own** ‚Äî add/reorder/duplicate; pick answer types (¬ß4.3); required/optional; **attach an image**
     to a question; add **simple branching** ("show Q/section if answer to Qx = value").
   - **A free-text brief** ‚Äî "what do you want to dive into?" ‚Üí AI drafts questions from it.
   - **From your data** ‚Äî AI generates from the **context-provider registry** (profiles, relationships,
     prior answers, Insights, recent sessions once `09` ships), **avoiding already-answered items** and
     **flagging stale data** worth refreshing. Output is schema-validated; a **safety pass** runs.
   - Per-question AI assists ("reword warmer", "make this for [person]").
3. **Preview / Test-on-self** ‚Äî see exactly what the recipient sees; optionally answer it yourself as a dry
   run (does **not** create coaching Insights).

A sent questionnaire is an **immutable snapshot** (¬ß4.2): editing a created questionnaire makes a new
version; in-flight assignments keep their original questions so answers always map.

### 3.2 Sending

1. **Choose recipient** ‚Äî a household **subject** (delivered in-app) or an **external person** (relay
   link). External requires `questionnaires.sendExternal` (default ON for Member) **and** a configured
   relay (¬ß3.8); without one, the sender is prompted to connect it (in-app/household sending still works).
2. **Privacy mode** ‚Äî _Standard_ (sender sees answers) or **Private** (break-glass, ¬ß8). For
   **compatibility** questionnaires, choose a **visibility mode** (¬ß3.6) ‚Äî this **drives the recipient's
   disclosure text** so the promise is always honest.
3. **Sender identity (external)** ‚Äî **named by default**, with a per-send **anonymous** option.
4. **Sensitive gating** ‚Äî Intimacy _General_ ‚Üí **18+ acknowledgement**; **Explicit / Unfiltered** ‚Üí
   **date-of-birth gate + explicit consent** on the recipient side (¬ß8.3).
5. **Link + delivery (external)** ‚Äî the app generates the relay link + a **6-digit PIN**, and offers
   **Copy link**, a prefilled **`mailto:`** (subject + body unique to this send; recipient email prefilled
   from the Person record), a prefilled **SMS** (`sms:` on mobile), and the **native share sheet**. The
   **PIN is included in the email + SMS by default** (per-send opt-out for sensitive sends). Default wording
   is the **`questionnaires.defaultMessages`** templates, editable in Settings (`{sender}`/`{link}`
   placeholders) and tweakable per send.

### 3.3 Recipient ‚Äî in-app (household)

Appears in the recipient's **Inbox** when they're the active person. They answer (save/resume), see the
**purpose, who's asking, the privacy mode, and a clear "Decline"** (silently, or with an optional short
note), then submit. Editable until submit, then locked.

### 3.4 Recipient ‚Äî external (relay page)

A standalone, branded, responsive, WCAG page served by the household's relay Worker (no app needed):

1. **Open link ‚Üí PIN gate.** The page requests the PIN before releasing any question content (the relay
   serves only ciphertext; the browser decrypts with the key in the URL **fragment**, which never reaches
   the server). Rate-limited (5 attempts ‚Üí 15-min lockout). Author-attached **question images** are
   likewise served as ciphertext and decrypted client-side (¬ß8.6).
2. **Consent / disclosure.** Shows purpose, **who's asking** (or "Someone" if anonymous), the **privacy
   statement derived from the configured visibility**, the **age gate** for sensitive tiers, the minimal
   **privacy notice**, **always-visible crisis resources + the not-medical line**, a quiet **"Sent securely
   via SelfOS"** trust line, and a clear **Decline** (silently, or with an optional short note).
3. **Answer.** Same renderer as in-app (one renderer, two hosts). Long forms **save + resume** (link +
   PIN). Answers are encrypted **in the browser** to the send's public key and POSTed as ciphertext.
4. **Return visit.** After submit, the link shows **"Thanks for filling this out."** A **withdraw** link
   deletes the response **before** it's drained; there is **no post-submit answer review** (SelfOS keeps no
   plaintext copy ‚Äî zero-knowledge). Expired/revoked links show **"This questionnaire is no longer
   available."**

### 3.5 Collection / drain

The app **drains** responses on open (and via a manual "Check for responses"); a "new responses" indicator
surfaces. Each is decrypted locally with the send's private key (¬ß5), validated, and the relay copy is
deleted on successful drain. Multi-device drains are idempotent.

### 3.6 Compatibility mode

Both people answer a per-answerer **personalized variant** of the same underlying questions. The author
**writes the canonical questions once**; AI **generates each answerer's personalized variant**, aligned by a
**canonical question id** (¬ß4.3), so the comparison stays apples-to-apples. AI then produces an alignment
report. The author picks a **visibility mode** per questionnaire ‚Äî and the recipient's disclosure follows it
automatically:

- **Shared report, raw hidden both ways** (default) ‚Üí recipient told _"Private‚Ä¶"_.
- **Sender sees all** ‚Üí recipient told _"Your answers will be shared with [sender]"_ (requires
  `questionnaires.readRaw`, ¬ß8).
- **Each sees own + shared report** ‚Üí recipient told their answers stay private to them; both see the
  joint report.

### 3.7 Analysis, Insights, metrics & the gap-finder

- **Analyze** ‚Äî analysis is **manual by default**: the sender taps **Analyze** on a response (a setting
  `questionnaires.autoAnalyze`, default **OFF**, can run it automatically on arrival). It's budget-gated
  (¬ß6) and turns answers into a structured **Insight**: a summary, key **facts**, and any **metrics** the
  questions declared (¬ß4.3 `metricKey`); a **crisis flag** is raised if answers disclose risk (¬ß8.2).
- **Approve-step** ‚Äî before the Insight enters `buildContext`, the sender reviews it in one pass: **edit the
  text**, **choose which facts are shareable** (¬ß4.4), or discard. Approved Insights stay editable/deletable
  later. Raw answers are never sent wholesale to the coach.
- **Insights surface ("what the coach knows")** ‚Äî list/edit/delete Insights with **provenance**; the
  shareable-vs-private split (`04`) governs whether a fact about a related person can feed **that** person's
  coaching.
- **Per-question trends (built ¬ß13.5c)** ‚Äî for numeric questions (rating/slider, and per row/bucket for
  matrix/allocation) the Results view charts each recipient's answers **over re-asks** (‚â•2 points), via core
  `buildQuestionTrends` + the `assignments:trends` IPC + the `LineChart` primitive. Trends include
  **Private** sends' numeric values (the disclosure is worded accordingly, ¬ß3.2/¬ß8.4) ‚Äî numbers only, never
  the prose answers.
- **Suggested (gap-finder)** ‚Äî AI scans **structured context only** (via the context-provider registry ‚Äî
  profiles, relationships, prior answers + Insights; **never raw chat transcripts**) and proposes the next
  questionnaire/questions. Budget-gated; over budget it renders a calm "budget reached" state without
  calling Claude.

### 3.8 Relay setup (admin-only Settings ‚Üí Relay)

Per-household **bring-your-own Cloudflare**, provisioned + deployed from the app:

1. **Connect** ‚Äî paste a **least-privilege Cloudflare API token** (Workers + KV/D1/R2 deploy scopes) +
   account id. The app **provisions** storage + **deploys** the zero-knowledge relay Worker to a free
   **`*.workers.dev`** subdomain (custom domain optional, later).
2. **Manage** ‚Äî status, **one-click update** when SelfOS ships a new relay version, and **teardown** on
   disable/uninstall.
3. **Config storage** ‚Äî endpoint, drain secret, and the **encrypted Cloudflare token** live in the **vault**
   (`config/relay.enc`, ¬ß4), so any household member sends/collects with **zero setup** and any admin
   device manages without re-entering the token.

### 3.9 Deletion

> **Build state (built ¬ß13.5c):** core `purgeQuestionnaire` (def + every send's snapshot/assignment/
> response + derived Insights ‚Äî and, for a compatibility send, the `questionnaires/compat/<groupId>/
report.enc` + the group's derived Insight, ¬ß13.5d) + `deleteSend` + `hasSends`; `Questionnaire.creatorPersonId` is main-stamped
> on create (preserved on edit, never back-filled onto a legacy def). `questionnaires:delete` is role-aware
> ‚Äî the Owner (`people.manage`) purges at any stage; a non-owner **creator** deletes their own
> **only while unsent**. Per-send delete is `assignments:delete` (sender/admin-only). Inline "Are you sure?"
> confirms in the builder + each Results send card. **Built ¬ß13.6:** the relay-link **revoke** before an
> external send's teardown, and **question-image garbage collection** ‚Äî `purgeQuestionnaire`/`deleteSend`
> (and an image-dropping save) run `imageGc.garbageCollectImages`, which deletes media referenced by no
> live definition or send snapshot (an image still frozen in a sent snapshot is kept until that send is
> also gone).

- **Owner** deletes **any** questionnaire at any stage ‚Üí **purge everything** (questionnaire +
  relay link + raw responses + Insights) after a clear confirmation; the relay link is revoked.
- **Creator** (incl. a Member) deletes **their own** questionnaire **only while unsent**, which also revokes
  any preview relay link.

## 4. Data model

All persisted formats are **Zod-backed** (`z.infer` types), versioned (`schemaVersion` + migrations), and
written through the vault service (`00` ¬ß4). Private content is encrypted (AES-256-GCM, master key ‚Äî `04`
¬ß5). Types live in `@selfos/core` so the renderer, IPC contract, and relay package share one source.

### 4.1 Vault layout (additions)

```
vault/
  config/
    relay.enc                      # encrypted: relay endpoint + drain secret + Cloudflare token
    questionnaires.json            # plain: default message templates, custom types, non-secret prefs
  questionnaires/
    defs/<questionnaire-id>.enc    # a created Questionnaire (versioned; re-sendable)
    media/<image-id>.enc           # encrypted author-attached question image (shared dir; isMediaPath-guarded)
    sends/<assignment-id>/
      questionnaire.enc            # immutable snapshot of the sent Questionnaire
      assignment.enc               # Assignment (status, channel, privacy, recipient, keys)
      response.enc                 # ResponseSet (encrypted answers) ‚Äî present once collected
      consent.enc                  # ConsentReceipt (external)
    compat/<group-id>/
      report.enc                   # AlignmentReport for a compatibility group (¬ß3.6/¬ß13.5d)
  people/<person-id>/insights/<insight-id>.enc          # Insight (source: questionnaire|session)
  relationships/<rel-id>/insights/<insight-id>.enc      # relationship-scoped Insight
```

**Multi-device / single-vault.** All questionnaire data ‚Äî created definitions, sends, responses, Insights,
and the relay config (incl. the encrypted Cloudflare token) ‚Äî lives **in the vault**, so every install
pointing at the same folder shares it; this feature stores **no per-device state**. Device-local `userData`
holds only the cross-cutting items owned by `00`/`04` (the keychain master key, window/session +
active-person state, logs). The **Owner is the full-access role** ‚Äî raw-answer access is permitted to the
Owner (any Private send) or a `senderSeesAll` sender holding `questionnaires.readRaw`; the previous
break-glass audit log was removed 2026-06-14 (per [`04-people-roles.md`](04-people-roles.md) ¬ß8).

### 4.2 Questionnaire & Question

```ts
type AnswerType =
  | 'shortText'
  | 'longText'
  | 'singleChoice'
  | 'multiChoice'
  | 'rating'
  | 'slider'
  | 'ranking'
  | 'thisOrThat'
  | 'yesNo'
  | 'date'
  | 'matrix'
  | 'allocation';

interface BranchRule {
  // simple conditional only (v1)
  whenQuestionId: string;
  equals: string | number | boolean;
  action: 'show'; // show this question/section when the rule matches
}

interface Question {
  id: string; // stable within a questionnaire
  canonicalId?: string; // shared across personalized variants (compatibility alignment)
  type: AnswerType;
  prompt: string;
  help?: string;
  required: boolean;
  media?: { imagePath: string; alt: string; mime: string }; // author-attached image (encrypted; ZK on relay)
  options?: string[]; // choice/ranking/thisOrThat/allocation buckets
  scale?: { min: number; max: number; minLabel?: string; maxLabel?: string; step?: number };
  matrix?: { rows: string[]; min: number; max: number; minLabel?: string; maxLabel?: string };
  metricKey?: string; // rating/slider/matrix ‚Üí populates Insight.metrics[metricKey] (¬ß4.4)
  branch?: BranchRule;
}

type SensitivityTier = 'standard' | 'intimacyGeneral' | 'explicit' | 'unfiltered';

interface Questionnaire {
  id: string;
  schemaVersion: number;
  version: number; // immutable-snapshot version
  creatorPersonId?: string; // main-stamped author (on create only); gates "creator deletes own while unsent" (¬ß3.9); legacy defs lack it ‚Üí Owner-deletable-only
  title: string;
  description?: string;
  type: string; // a starter-taxonomy key OR a user-defined custom type
  sensitivity: SensitivityTier;
  questions: Question[];
  compatibility?: { enabled: true; visibility: 'sharedReport' | 'senderSeesAll' | 'eachSeesOwn' };
  createdAt: string;
  updatedAt: string;
}
```

`allocation` questions use `options` as the buckets (answers must sum to 100). `matrix` rates each
`matrix.rows` item on the one shared scale. Custom types are free-text strings persisted in
`questionnaires.json` so they reappear in the type picker.

### 4.4 Insight (the shared Insight / metrics layer)

```ts
type InsightSource = 'questionnaire' | 'session' | 'dream'; // 'session' by spec 09, 'dream' by spec 12

interface InsightFact {
  id: string;
  text: string;
  shareable: boolean; // false = private to the subject; true = may feed related people
}

interface Insight {
  id: string;
  schemaVersion: number;
  source: InsightSource;
  subjectPersonId: string; // whose coaching this informs
  relationshipId?: string; // relationship-scoped insights
  summary: string;
  facts: InsightFact[];
  metrics?: Record<string, number>; // named normalized signals (chartable; e.g. connection, desire,
  //   moodValence, moodEnergy). The flexible basis for all trends.
  confidence: 'low' | 'medium' | 'high';
  approved: boolean; // questionnaire insights require approval; session insights auto-true (09)
  provenance: {
    assignmentId?: string;
    conversationId?: string;
    dreamId?: string; // spec 12
    compatibilityGroupId?: string; // ¬ß13.5d: a compatibility alignment Insight
    at: string;
  };
  crisisFlag?: boolean;
  createdAt: string;
  updatedAt: string;
}
```

`metrics` is the single, extensible basis for every trend in the app: questionnaire answers populate it via

## 7. States & edge cases

Per `00` ¬ß7, every surface handles loading / empty / error / offline. Specifically:

- **No Claude key / AI disabled** ‚Äî manual authoring + sending still work; generation/analysis/gap-finder
  show calm "enable AI" states; no Insights produced.
- **Over budget** ‚Äî generation/analysis/gap-finder block with the `06` warn‚Üíblock UX (owner override); the
  gap-finder degrades to "budget reached" without calling Claude.
- **No relay configured** ‚Äî in-app/household sends work; external send is offered with a "connect a relay"
  prompt; never an error.
- **Relay deploy failure / token invalid / quota** ‚Äî typed, actionable errors; partial provisioning is
  idempotently retried; nothing half-written claims success.
- **PIN brute-force** ‚Äî lockout after 5 attempts/15 min; the sender can reset/resend the PIN.
- **Expired / revoked / already-submitted link** ‚Äî neutral "no longer available" / "thanks" states.
- **Lost device before drain** ‚Äî responses recoverable (the send private key is wrapped under the master
  key in the vault, recovery-phrase recoverable).
- **Edited/deleted definition with in-flight sends** ‚Äî sends use their immutable snapshot; unaffected.
- **Sync conflict** on a questionnaire/Insight/relay file ‚Äî vault conflict detection (`00`); never
  auto-deleted; the `relay.enc` token conflict is surfaced, not overwritten.
- **Schema migration** ‚Äî `Person` (email/phone) + any questionnaire/Insight bumps migrate on read.
- **Crisis content in a free-text answer** ‚Äî analysis raises `crisisFlag`; the result **leads with
  resources** (¬ß8.2).

## 8. Safety, privacy & honesty

### 8.1 Wellness boundary

SelfOS remains **wellness/self-help, not medical** (CLAUDE.md ¬ß1). Questionnaires never diagnose, score
diagnostically, or reproduce copyrighted/clinical instruments ‚Äî "science-informed" means **original,
evidence-informed** questions in the SelfOS voice. The not-medical line is visible in-app and on the relay.

### 8.2 Crisis routing

The relay page and the Inbox always show **curated crisis resources + "Get help now"** (static; no Claude on
the relay). During analysis a model-based **crisis flag** (no keyword interstitial, consistent with `05`)
makes the result **lead with concern + resources**. For external recipients SelfOS can't intervene live ‚Äî
the always-present static resources are the safeguard.

### 8.3 Sensitive content & age

Intimacy _General_ ‚Üí **18+ acknowledgement**; **Explicit / Unfiltered** ‚Üí **date-of-birth gate + explicit
consent** before any explicit content (incl. question images) renders; a **ConsentReceipt** is stored with
the response; household recipients use the birthday on their Person record where present. All
generation/analysis runs **within Anthropic's usage policy** ‚Äî refusals handled gracefully, never
circumvented. **iOS/App-Store risk** for explicit tiers is recorded (¬ß11) with mitigations (17+ rating, the
gate, store-guideline compliance, remote disable).

### 8.4 The private-mode privacy model

- **Promise to the recipient:** in **Private** mode the recipient is told their answers are _"used only to
  personalize coaching."_ In normal operation that is true: **no UI exposes the raw/prose answers**, and the
  `questionnaires.readRaw` capability ships **OFF for non-owner roles**. _Since ¬ß13.5c the disclosure also
  states that the recipient's **numeric ratings may appear in the sender's trends over time** (the
  rating-over-time charts include Private sends' numbers ‚Äî never the prose answers); the copy is kept
  consistent across the send panel + the recipient Inbox so the promise stays honest (¬ß3.2)._
  `questionnaires.readRaw` is an **explicit-grant-only** capability: it ships OFF for non-owner roles and is
  enabled only by an explicit toggle in the Roles matrix. The **Owner is the full-access role and always
  holds it.**
- **Raw answers can be revealed two ways** (each via `assignments:revealRaw`, which writes **no audit
  entry** ‚Äî the break-glass audit log was removed 2026-06-14): the **Owner** (any Private send), or the
  **sender of a `senderSeesAll` compatibility send** holding `questionnaires.readRaw` (¬ß3.6/¬ß13.5d).
  Nothing else exposes the prose answers.
- **We never surface owner/admin visibility to answerers** (a durable product rule, 2026-06-15). The
  `questionnaires.discloseAdminAccess` setting + the `ADMIN_ACCESS_DISCLOSURE` line were **removed** (¬ß16.6):
  no recipient is ever told a household owner/administrator could access their answers, and no author-facing
  copy mentions it either.
- **Honesty guard** ‚Äî the recipient's disclosure text is **derived from the configured visibility/privacy
  mode**, so no configuration can promise more privacy than the system delivers. (It states what the
  **sender** sees ‚Äî never owner/admin reach.)
- **Not zero-knowledge from the owner** ‚Äî consistent with `04` ¬ß8 and `10-multi-device-vault` ¬ß2/¬ß8 (one
  master key decrypts the whole vault; RBAC is enforced at the app/UX layer, not by encryption).

### 8.5 Coercion & consent

Every send shows the recipient the **purpose, who's asking, the privacy mode, and a frictionless Decline**;
sensitive sends emphasize they may skip/stop; no dark patterns. External recipients can **withdraw/delete**
before drain.

### 8.6 Relay & data handling

Zero-knowledge: the Worker stores **ciphertext only**; questions **and author-attached images** are
decrypted via the URL **fragment** key (never seen by the server); responses are encrypted to the send public
key in the browser. A **minimal privacy notice** (what's stored, that it's encrypted + unreadable to the
host, retention, how to delete) is served on the relay ‚Äî required despite zero-knowledge because
IP/timestamps/payloads are still processed (GDPR/CCPA + App Store). The Cloudflare token is **least-privilege**
and **encrypted** in the vault.

## 15. 2026-06 amendment ‚Äî authoring-experience refinements

Layers on the built feature (¬ß1‚Äì¬ß14 remain accurate). Covers app-refresh items **6** (General type +
intimacy/scenario-only sensitivity), **7** (clearer "who sees what" copy), **8** (drop the "Use my information" toggle),
and **9** (live inline question preview). Mostly renderer; the only schema-adjacent change is one taxonomy
entry and where sensitivity is surfaced. No new capabilities, no new IPC channels.

### 15.1 A "General" type (item 6)

Add **`general`** ("General") to the starter taxonomy (¬ß3.1) ‚Äî a catch-all for a questionnaire that isn't tied
to a specific frame (role-feedback, blind-spots, intimacy, ‚Ä¶). It becomes the **default selected type** for a
new questionnaire (today the builder defaults to whatever is first; General is the most neutral starting point).
Like every type it carries **no pre-authored content** and simply steers generation/categorization. No schema
change ‚Äî `Questionnaire.type` is already a free string keyed off the starter list + custom types; `general` joins
the starter list (single source of truth shared by the type `<select>` and the gap-finder).

### 15.2 Sensitivity is intimacy-only (item 6)

Today the **Sensitivity** picker shows the same four intimacy-flavoured options
(`standard` / `intimacyGeneral` / `explicit` / `unfiltered`) on **every** type, which reads as out of place on,
say, a role-feedback questionnaire. New rule:

A small **`SENSITIVITY_TYPES`** constant (single source of truth) declares which types surface the picker, with
two behaviours:

- **`intimacy`** ‚Äî always sensitive. The picker shows the **intimacy tiers only**
  (`intimacyGeneral` ¬∑ `explicit` ¬∑ `unfiltered`), default `intimacyGeneral`; **no `standard` option** (an
  intimacy questionnaire is at least 18+ general). Switching **to** intimacy seeds `intimacyGeneral`.
- **`scenario`** ‚Äî _may_ be sensitive (a scenario can be sexual, or entirely tame ‚Äî work, family). The picker
  shows with **`standard` as the default** plus the intimacy tiers available to escalate, so a sexual scenario
  can be marked explicit while a work scenario stays standard.
- **All other types** ‚Äî sensitivity is **implicitly `standard`**, the picker is **hidden**, and the value is
  forced to `standard` on save. Switching a questionnaire's type to one **not** in `SENSITIVITY_TYPES` resets
  sensitivity to `standard`.

The `SensitivityTier` enum (¬ß4.2) is **unchanged**; this is purely _when_ the control is shown and what default
it carries. All the existing send-time age/consent gating (¬ß8.3) is unaffected ‚Äî it already keys off the tier,
which only `intimacy`/`scenario` questionnaires can now carry as non-`standard`.

### 15.3 Clearer "who sees what" copy (item 7)

The **compatibility visibility** picker (¬ß3.6) keeps its three modes (`sharedReport` / `eachSeesOwn` /
`senderSeesAll`) and the whole privacy/audit mechanism (¬ß8.4) ‚Äî only the **author-facing labels + help** change,
to drop jargon the user flagged ("break-glass", "audited", the redundant-sounding "each sees their own"):

| --------------- | ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------- |
| `sharedReport`  | **Shared report only**               | "Neither of you sees the other's answers ‚Äî you both get one combined report."                                                               |
| `eachSeesOwn`   | **Shared report + your own answers** | "You both get the combined report, and each person can also look back at their **own** answers. Neither sees the other's."                  |
| `senderSeesAll` | **You see their answers**            | "You'll see [recipient]'s individual answers and you both get the combined report. They're clearly told their answers are shared with you." |

### 15.4 Remove the "Use my information" AI toggle (item 8)

### 15.5 Live inline question preview (item 9)

data** (answering it does nothing, like the existing full Preview mode's dry-run, ¬ß3.1).
This complements ‚Äî doesn't replace ‚Äî the existing whole-questionnaire **Preview / Test-on-self** mode (¬ß3.1): the
inline preview is per-question while authoring; the full mode is the end-to-end recipient experience.

## 16. 2026-06-15 amendment ‚Äî audit fixes & enhancements

A full audit of the built feature (the compatibility send flow, the authoring workflow, AI generation by
tier, and test coverage) found three real problems plus the three enhancements the user asked for. This
section specifies all of them. Layers on ¬ß1‚Äì¬ß15 (which remain accurate). All decisions were asked-first.

### 16.1 Compatibility participant model ‚Äî the sender can be a participant (the core fix)

**Problem.** Compatibility today assumes the **sender is a neutral third party** comparing **two other**
people (the bridge rejects the sender being a participant ‚Äî "Send a compatibility check to two other
people."). But the obvious couples case is "compare **me** with my partner." A user authoring "Sexy Time"
and picking themselves + their partner hits a confusing late rejection, and the disclosure copy
(`compatibilityDisclosure(visibility, senderName)`) names the **sender** as one of the two answerers
("neither you nor [sender]‚Ä¶"), which is also shown to the recipients ‚Äî wrong on both counts.

**Resolution (decision: "you + someone else", keep two-others).** Compatibility supports **both**:

- **You + someone else** (the primary, default case): the **sender is one of the two participants**; the
  other person answers their personalized variant; the report/insight follows the visibility mode.
- **Two other people** (retained): the sender compares two others and is the report's audience.

Changes:

- **Backend** ‚Äî `createCompatibilitySend` + the bridge guard accept the sender as a participant; the
  invalid case becomes "the **same** person picked twice," not "sender is a participant." The two paired
  assignments are built for the two **participants** (which may include the sender). `alignmentService`
  aligns the two participants (not "two others"); insight routing per ¬ß16.2.
- **Disclosure (`disclosure.ts`)** ‚Äî `compatibilityDisclosure` is reworked to describe the **two actual
  participants by the relationship in play**, never the sender-as-third-party. When the sender is a
  participant it reads naturally ("Your individual answers stay private ‚Äî neither you nor **Angel** sees
  the other's; you'll both get one combined report."). The function takes the **participant context** it
  needs (e.g. the other participant's name + whether the viewer is a participant) instead of just
  `senderName`. The honesty guard (same text shown to recipients) is preserved.
- **UI (`CompatibilitySendPanel`)** ‚Äî relabel to "**You + someone else**" vs "**Two other people**"
  (a small mode toggle, defaulting to you+someone-else with the sender pre-selected as the first
  participant); the picker disables only the already-chosen person; the live disclosure preview reflects
  the real participants. The invalid-send state is now unconstructable in the common path.

### 16.2 Context-only visibility mode (new) ‚Äî no report, feeds both contexts

**Decision.** Add a **fourth** compatibility visibility option,
`CompatibilityVisibility += 'contextOnly'`: both people answer their variants, but **no report is generated
or shown to anyone and no raw answers are shared** ‚Äî the answers are distilled into Insight(s) that
**inform both participants' own coaching contexts**. Recipients are told exactly this (a derived disclosure
line). It is the most private mode (analysis with zero human-facing output).

- **Analysis** ‚Äî `alignmentService` (or a sibling) runs the distillation but produces **no `AlignmentReport`
  surface**; instead it writes a distilled, own-context Insight for **each participant** (their own coaching
  context is enriched by the compatibility answers; one participant's **raw** answers are never exposed to
  the other ‚Äî only the distilled understanding). Facts default own-context-only (no cross-person raw share),
  consistent with [`15`](15-shareability.md). Budget-gated like the existing analyze (¬ß3.7).
- **Picker** ‚Äî `contextOnly` joins the ¬ß15.3 "who sees what" list with plain copy: _"No report ‚Äî answers
  just help each of you be better understood by your own coach."_ No `readRaw` needed (nothing is revealed).

### 16.3 Create ‚Üí Send: a clear two-step workflow

**Problem.** The builder shows **Create/Save** and **Send** side-by-side, both gated by the same `canSave`,
both terminal ‚Äî but "Create" saves **and closes the builder** (so you can't Send after it) while "Send"
**auto-saves internally** then opens the send panel. "Create then Send" looks like a sequence but isn't.

**Resolution (decision: "Save draft ‚Üí Send from the saved questionnaire").**

- While editing, the primary action is **Save** (creates/updates the draft and **keeps you on the saved
  questionnaire**, not closing the builder). A clear saved/dirty state distinguishes "new draft" from
  "saved."
- **Send** is a distinct step available **on a saved questionnaire** (its own button on the saved
  detail/list row), not a co-equal of Save in the edit footer. Sending still validates first; if the draft
  has unsaved edits it prompts/saves them. This makes the flow read **create ‚Üí then send**.
- Remove the strand where Create closes the builder and leaves no path to Send.

### 16.4 Title below "Draft with AI" + an AI-generated title

**Decision.** Move the **Title** field from the top card to **below the `QuestionnaireAiPanel` ("Draft with
AI") and above the first question**, and have AI generation **also produce a title**:

- `QuestionnaireGenerateResult` gains an optional **`title?: string`**; `generationService` asks for a short
  title alongside the questions; the IPC result carries it.
- `QuestionnaireAiPanel` surfaces the title (e.g. an `onTitle` callback or via `onGenerated`); the builder
  sets `title` from it **only when the title field is empty** (never clobbers author input).
- The builder layout detaches Title from the Type/Sensitivity/Compatibility card and re-places it after the
  AI panel.

### 16.5 Tier-distinct explicit generation (within the consensual-adult boundary)

**Problem.** Generation is **tier-blind**: `GENERATION_SYSTEM` never mentions the tier and is conservative
("supportive voice‚Ä¶ stay strictly within policy‚Ä¶ return [] if in doubt"); `explicit` and `unfiltered` map to
**byte-identical, tame** one-liners in `SENSITIVITY_NOTE`. Nothing ever **instructs** explicitness, so the
model defaults to tasteful ‚Äî the user's "unfiltered" produced emotional-closeness questions.

**Resolution (decision: match the intake bar; distinct tiers).** Bring questionnaire generation up to the
explicitness the **intake** intimacy block already ships, within the **same** consensual-adult boundary:

- In `packages/core/src/questionnaires/aiPrompts.ts`, make `explicit` and `unfiltered` **distinct** and give
  each **positive, escalating** content direction. For the **intimacy** type at these tiers (18+),
  generation is **instructed** to write genuinely explicit, specific questions ‚Äî acts, body/grooming,
  fantasies (incl. taboo fantasies **as roleplay**, e.g. CNC framed as pre-agreed), porn, history ‚Äî with
  **`unfiltered` the most graphic / no-holds-barred within the boundary** and `explicit` a notch below.
- Add a **tier-aware intimacy generation framing** (a dedicated system clause or string used when the tier
  is `explicit`/`unfiltered`) that **positively permits and requests** explicit content ‚Äî rather than the
  current refusal-default ‚Äî and seed the user message with an in-policy topic inventory (mirroring the
  intake catalog's `ACTIVITIES`/`commonFantasies`), so the model has concrete subject matter.
- **The boundary is unchanged and explicit in the prompt** (CLAUDE.md; the intake `contentNote`):
  consensual-adult sexuality only; taboo **fantasies** in as fantasy/roleplay with advance consent; **never**
  minors, real non-consent, or illegal acts; stays within Anthropic's usage policy (refuse ‚Üí `[]` only when
  genuinely out of policy, not merely because it's explicit). The shared `SAFETY` prefix used by the
  gap-finder/analysis/variant/alignment prompts is **not** loosened globally ‚Äî only the intimacy-generation
  path gains the explicit framing. The 18+/consent gate stays recipient-side at send (¬ß8.3).

### 16.5a Owner-extensible intimacy topic inventory (shared with the intake)

The in-policy topic inventory that seeds generation (and that the **intake** intimacy block already uses as its
`ACTIVITIES`/`commonFantasies` lists, [`18`](18-personal-onboarding.md)) becomes **one shared, owner-extensible
source of truth**:

- **Shared built-in constant** ‚Äî extract the built-in activities + fantasies into a single shared constant
  (e.g. `@selfos/core` `INTIMACY_TOPICS = { activities: [...], fantasies: [...] }`) that **both** the intake
  catalog and questionnaire generation import. (Resolves ¬ß16.9.2 ‚Äî one source of truth, no drift.)
- **Owner-managed custom additions** ‚Äî the Owner can **add/remove custom `activities` and custom `fantasies`**
  (mirroring the two built-in lists), persisted **vault-side, household-wide** (a new prefs file, the
  `config/questionnaires.json` custom-types precedent). The **merged** inventory (built-in + custom) is what
  feeds **both** the intake intimacy form and questionnaire generation.
- **Two entry points (decision)** ‚Äî a dedicated **owner-only Settings** surface (18+, marked, vault-scoped) to
  manage the lists, **and** a quick **inline "add a topic"** in the intimacy questionnaire builder that writes
  to the same shared custom list. Both edit one inventory.
- **Boundary (decision: trust owner + model-enforced policy + a clear note)** ‚Äî additions are free text; the
  Owner is the full-access role, so additions aren't keyword-filtered. The generation prompt + Claude still
  enforce Anthropic's usage policy and the consensual-adult boundary (minors/real-non-consent/illegal never
  become activities, even if typed). A clear note states additions must be **consensual-adult and within
  policy**. Custom topics inherit the same 18+/restricted handling as the built-ins.
- **Cross-spec** ‚Äî this touches [`18`](18-personal-onboarding.md): the intake intimacy block reads the **merged**
  inventory instead of a hard-coded local list (sync 18 when built).

### 16.7 Tests ‚Äî the comprehensive questionnaire E2E coverage matrix

The audit found strong unit coverage but **E2E gaps** that let a whole-feature model mismatch (the
compatibility participant bug) ship unnoticed. This amendment establishes an **exhaustive, start-to-finish
E2E regression suite** covering **every option/scenario** of the questionnaire feature ‚Äî authoring ‚Üí sending
‚Üí answering ‚Üí viewing/results ‚Üí deletion ‚Äî so a future regression in any path is caught. **Goal: every
option exercised end-to-end at least once, plus the important cross-products** (not the full cartesian
explosion ‚Äî e.g. each answer type is covered once, and each compatibility visibility √ó the
send‚Üíanswer‚Üíreport flow is covered, but not every answer-type√ótype√ótier combination). Where a real browser
relay round-trip isn't feasible, drive the recipient side via the relay test utilities (the existing
pattern). All Claude/image/relay calls use the established fakes (`SELFOS_FAKE_CLAUDE`/`SELFOS_FAKE_IMAGE`/
`SELFOS_FAKE_RELAY`); decrypt the real vault to assert data, not just the UI. Run `pnpm typecheck` after
tests (memory `vitest-does-not-typecheck`).

**A. Authoring (each exercised E2E):**

- Each **answer type** authored + answered + round-tripped: shortText, longText, singleChoice, multiChoice,
  rating, slider, ranking, thisOrThat, yesNo, date, matrix, allocation (incl. allocation-sums-to-100 + the
  scale min/max + matrix rows + help text).
- Each **questionnaire type** selectable incl. **General** default; **custom type** add ‚Üí persists across an
  app restart ‚Üí reappears in the picker; duplicate custom type shows the ¬ß16.6 inline notice.
- **Sensitivity** picker shown only for intimacy/scenario; hidden + forced `standard` elsewhere; each
  intimacy tier (`intimacyGeneral`/`explicit`/`unfiltered`) selectable.
- **Branching** (show-when), the **inline per-question preview**, **question images** (attach + alt required
  - **size >5 MB rejected** + **wrong MIME rejected** + encrypted round-trip + display).
- **AI draft** (mocked): generates **questions + a title** (title fills only when empty); **per-question
  improve**; **gap-finder/Suggested** with a mock; plus the calm AI-off / over-budget / no-key states.
- **Validation**: each invalid state (missing options, bad branch target, empty matrix, NaN scale) **blocks
  Send with a specific message** (¬ß16.6).
- **Create ‚Üí Send workflow**: Save a draft ‚Üí it **stays saved** (no strand) ‚Üí Send is a **separate step** on
  the saved questionnaire (¬ß16.3).

**B. Sending (each channel √ó each mode):**

- **In-app**, each **privacy** mode: `standard` (sender sees answers) and `private` (break-glass / analyze-
  only) ‚Äî assert the recipient is shown the correct **disclosure** text for each.
- **Compatibility**, each **participant model** (`you + someone else`, `two others`) √ó each **visibility**
  (`sharedReport`, `eachSeesOwn`, `senderSeesAll`, **`contextOnly`**) ‚Äî at minimum each visibility once end-
  to-end, and both participant models. Assert the reworked **`compatibilityDisclosure`** names the real two
  participants and **never** the sender as the "other"; assert `senderSeesAll` is gated by `readRaw`.
- **External/relay**: mint link + PIN; each **delivery** option (copy/mailto/SMS/share); **named vs
  anonymous** sender; the **18+/DOB+consent** gate for explicit/unfiltered tiers on the relay page.

**C. Answering (recipient):**

- Inbox receive ‚Üí answer ‚Üí **save progress ‚Üí resume** ‚Üí submit; **decline** (silent + with note); **required**
  gating; **branching** visibility on the answer side.
- **Relay page**: unlock with PIN ‚Üí decrypt ‚Üí answer ‚Üí seal ‚Üí sender **drains** ‚Üí ResponseSet persisted
  (decrypt to verify); **wrong PIN** lockout; **revoked/expired** token denied; drain with **zero responses**
  and **decline-only**.

**D. Viewing / Results / Analysis:**

- `standard` send ‚Üí **raw answers shown**; `private` send ‚Üí **raw hidden**, Analyze-only.
- **Analyze** ‚Üí draft Insight ‚Üí **approve in Memory** ‚Üí it **feeds a later session's `buildContext`**
  (decrypt to assert); **autoAnalyze** setting on ‚Üí auto-runs.
- **Compatibility**: both submit ‚Üí **alignment report** + per-question agreement; **regenerate** reuses the
  Insight; `senderSeesAll` + `readRaw` ‚Üí **reveal raw**; **`contextOnly`** ‚Üí **no report produced** and
  **both** participants' own contexts enriched per-participant (decrypt to assert, no cross-exposure).
- **Trends**: rating/slider re-asks chart (‚â•2 points), **multi-recipient** aggregation.

**E. Deletion / lifecycle:**

- Delete a **single send** ‚Üí its Insight cleaned up; **purge a whole questionnaire** ‚Üí all sends + Insights +
  orphan images cascade; delete a **compatibility member** ‚Üí the group report + group Insight torn down;
  **relay revoke** (manual + revoke-on-delete).

**F. Gating / privacy / cost:**

- Each capability gates the right surface: `create`, `answer`, `viewResults`, `sendExternal`, and
  `readRaw` (EXPLICIT_GRANT_ONLY ‚Äî `senderSeesAll` picker + reveal). A member **without** `sendExternal`
  can't open the relay send; the **admin-access disclosure** setting flips the recipient disclosure;
  **cost/$ stays admin-only**.

**G. Failure / offline:**

- AI off / no key / over budget ‚Üí calm states across builder + Suggested + Analyze (no dead buttons);
  corrupt/malformed ResponseSet handled (skips unknown question ids, never throws).

**Unit/component** still back the matrix (the reworked participant model, `contextOnly` per-participant
insight routing, the tier-distinct explicit framing, disclosure strings) ‚Äî E2E proves the wiring, units prove
the logic. This matrix is the **standing regression suite**; new questionnaire options must extend it (a DoD
note for future questionnaire work).
